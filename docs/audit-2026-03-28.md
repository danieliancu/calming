# Audit Calming

Data: 2026-03-28

## Rezumat executiv

Aplicatia are o baza functionala buna pentru un produs Laravel + Inertia cu mai multe roluri, dar exista cateva riscuri reale pentru productie:

- un endpoint de debug expus public care returneaza date de sesiune si thread-uri assistant;
- protectii inconsistente la nivel de rute pentru zone privilegiate;
- configuratie sensibila cu impact direct asupra confidentialitatii si operarii (`SESSION_ENCRYPT=false`, toate mecanismele stateful in DB);
- zone evidente de N+1 si bootstrap repetitiv in dashboard-uri si profile;
- acoperire SEO incompleta: lipsesc canonical, sitemap, metadate consistente pentru paginile publice si exista probleme de encoding in head.

Separat, suita de teste ruleaza in mare parte bine, dar are 3 regresii deja vizibile in `tests/Feature/ProfileTest.php`, ceea ce reduce increderea in traseele de profil/account.

## Context verificat

- Stack: Laravel 12, Inertia React 18, Vite.
- Config principal: MySQL, `SESSION_DRIVER=database`, `CACHE_STORE=database`, `QUEUE_CONNECTION=database`.
- Suprafata aplicatie: 122 rute aplicative.
- Roluri: user, psychologist, superadmin.
- Pattern dominant de acces la date: `DB::table(...)`, cu utilizare limitata a modelelor Eloquent pentru agregare si eager loading.
- Verificari rulate:
  - `php artisan route:list --except-vendor`
  - `php artisan test --parallel`
  - inspectie statica a config-ului, migrarilor, controllerelor API/web si a setup-ului SEO.

## Findings

### Critic

#### 1. Endpoint de debug expus public cu date de sesiune si assistant

- Dovezi:
  - `routes/api.php` expune `GET /api/debug/assistant` doar in middleware-ul `web`
  - `app/Http/Controllers/Api/DebugAssistantController.php` returneaza `session()->all()`, `session_id`, statusul tabelelor si `assistant_threads`
- Impact:
  - expunere directa de date sensibile din sesiune;
  - ajuta enumerarea mediului si a structurii aplicatiei;
  - poate divulga context personal al utilizatorului autentificat.
- Scenariu realist:
  - un atacator sau un user curios poate interoga endpoint-ul in productie si extrage date utile pentru escaladare sau profiling.
- Recomandare:
  - elimina complet endpoint-ul din productie sau restrictioneaza-l strict la `local/testing`;
  - daca ramane, protejeaza-l cu auth + ability explicita si nu returna niciodata `session()->all()`.

### High

#### 2. Securitatea de acces pentru zone privilegiate este implementata in controller, nu in middleware de ruta

- Dovezi:
  - `routes/web.php` expune rutele `superadmin/*` si `psychologists/dashboard/*` fara grupuri middleware dedicate;
  - `SuperadminController` foloseste `requireSuperadminSession(...)`;
  - `PsychologistController` foloseste validare in metode precum `dashboard(...)`.
- Impact:
  - model de securitate usor de ocolit accidental la adaugarea unor rute noi;
  - protectia devine dependenta de disciplina implementarii, nu de infrastructura de routing.
- Scenariu realist:
  - o metoda noua adaugata in controller poate omite check-ul manual si expune operatii administrative.
- Recomandare:
  - introduce middleware dedicate pentru `superadmin` si `psychologist+mfa-fresh`;
  - grupeaza toate rutele sensibile sub aceste middleware.

#### 3. Lipsa rate limiting pentru fluxuri cu potential de abuz

- Dovezi:
  - `routes/api.php` nu aplica `throttle` pe `POST /api/assistant/guest/respond`;
  - rutele psychologist login/MFA/resend din `routes/web.php` nu au throttle dedicat;
  - singurele throttle explicite sunt pe verificarea email standard din `routes/auth.php`.
- Impact:
  - abuz de cost pentru providerul assistant;
  - brute-force pe autentificarea psychologist/MFA resend;
  - amplificare de incarcare pe DB deoarece sesiune/cache/queue sunt tot in DB.
- Recomandare:
  - adauga rate limits pe IP + sesiune pentru assistant guest;
  - adauga throttle pe signin psychologist, resend verification si resend MFA.

#### 4. Documentele de validare ale psihologilor pot deveni public accesibile prin URL direct

- Dovezi:
  - `SuperadminController.php` construieste `Storage::disk($document->disk)->url($document->path)` pentru documentele incarcate.
- Impact:
  - daca disk-ul este public sau configurat cu acces direct, documente sensibile pot fi accesate prin URL predictibil sau redistribuit.
- Recomandare:
  - muta documentele pe disk privat;
  - serveste-le prin controller autorizat, cu `temporaryUrl` doar unde suportul o permite.

#### 5. Configuratie cu risc pentru confidentialitate si operare

- Dovezi:
  - `.env.example`: `SESSION_ENCRYPT=false`
  - `.env.example`: `SESSION_DRIVER=database`, `CACHE_STORE=database`, `QUEUE_CONNECTION=database`
  - mediul local prezent in repo indica si `APP_DEBUG=true`
- Impact:
  - sesiuni si stare operationala depind de aceeasi baza de date;
  - datele de sesiune sunt pastrate necriptate la nivel de aplicatie;
  - in productie, setarile gresite pot expune stack traces si date sensibile.
- Recomandare:
  - forteaza `APP_DEBUG=false` in productie;
  - activeaza criptarea sesiunii sau muta sesiunea pe storage mai potrivit;
  - separa macar cache/queue de DB in productie, ideal spre Redis.

#### 6. Integritatea verificarii email pentru psihologi a fost slabita la migrare

- Dovezi:
  - `database/migrations/2026_03_16_130000_add_psychologist_security_controls.php` seteaza `email_verified_at = now()` pentru toti psihologii unde era null.
- Impact:
  - conturi vechi sunt tratate ca verificate fara dovada reala;
  - reduce valoarea fluxului de verificare email introdus ulterior.
- Recomandare:
  - trateaza aceasta migrare ca bootstrap istoric, nu ca politica permanenta;
  - ruleaza o revalidare controlata pentru conturile migrate sau marcheaza explicit originea validarii.

#### 7. Dashboard-urile administrative contin N+1 evidente si query-uri repetitive

- Dovezi:
  - `SuperadminController.php`: in map se fac query-uri suplimentare pentru attestations, specializations, documents, messages si statistici per template;
  - `PsychologistController.php` dashboard compune date din multiple query-uri separate si map-uri cu query-uri interne.
- Impact:
  - degradare vizibila de performanta cand cresc aplicatiile, documentele, template-urile sau istoricul.
- Recomandare:
  - preincarca datele pe loturi;
  - muta agregarile in query-uri dedicate;
  - evita `count()`/`first()`/`pluck()` in bucle.

### Medium

#### 8. Bootstrap-ul profilului foloseste prea multe query-uri mici si raw SQL interpolat

- Dovezi:
  - `app/Support/UserProfileBootstrapper.php` face `value()` repetate pe `user_profiles` si `user_profile_details`;
  - `syncStats()` construieste query-uri `DB::raw` cu interpolarea directa a `userId`.
- Impact:
  - supracost la fiecare `ensureForUser(...)`;
  - greu de optimizat si testat;
  - pattern periculos daca interpolarea ajunge vreodata pe input neincredibil.
- Recomandare:
  - citeste fiecare row o singura data;
  - parametrizare in loc de interpolare;
  - cacheaza stats unde nu trebuie recalculare imediata.

#### 9. Feed-ul de notificari este incarcat integral si apoi sortat in memorie

- Dovezi:
  - `NotificationService::feedFor()` ia toate notificarile publice si personale, apoi concateneaza si sorteaza in Collection;
  - `NotificationBootstrapController` apeleaza bootstrap-ul frecvent.
- Impact:
  - cost in memorie si query time proportional cu volumul istoric.
- Recomandare:
  - limiteaza feed-ul;
  - ordoneaza si filtreaza mai agresiv in SQL;
  - adauga paginare sau fereastra temporala.

#### 10. Home si Community muta multa logica de agregare in PHP dupa query-uri late

- Dovezi:
  - `CalmPageController::home()` si `::community()` trag seturi mari de randuri si grupeaza/sorteaza in PHP.
- Impact:
  - cost inutil la cresterea datelor;
  - bootstrap Inertia mai mare decat necesar.
- Recomandare:
  - muta agregarile catre SQL sau materializeaza snapshot-uri;
  - returneaza doar top-N relevant direct din query.

#### 11. Nu exista politica clara de retentie pentru date sensibile

- Dovezi:
  - nu exista job-uri sau comenzi evidente pentru curatarea `assistant_messages`, `psychologist_mfa_challenges`, `psychologist_email_verifications`, `notifications`, `sessions`.
- Impact:
  - crestere continua a datelor;
  - risc de confidentialitate si cost operational;
  - baza devine si mai critica pentru starea aplicatiei.
- Recomandare:
  - defineste ferestre de retentie;
  - adauga job-uri programate pentru purge/anonymize/archive.

#### 12. Acoperirea testelor este rezonabila, dar exista regresii deschise pe profil/account

- Dovezi:
  - `php artisan test --parallel` a trecut 50 teste si a picat 3 in `tests/Feature/ProfileTest.php`;
  - toate cele 3 arata drift intre `/account` si ruta canonica de profil.
- Impact:
  - fluxul de profil are comportament nealiniat cu asteptarile suitei;
  - semnal de fragilitate in routing/redirect logic.
- Recomandare:
  - decide forma canonica a rutei si actualizeaza fie codul, fie testele;
  - trateaza aceste esecuri ca regression debt, nu ca zgomot.

### SEO High

#### 13. Head-ul global are probleme de encoding si poate produce snippet-uri defecte

- Dovezi:
  - `resources/views/app.blade.php` contine texte romane cu mojibake in `meta description`, `og:title`, `twitter:title`.
- Impact:
  - snippet-uri stricate in Google/social;
  - semnal slab de calitate si brand.
- Recomandare:
  - normalizeaza fișierul la UTF-8 fara BOM;
  - reverifica tot textul romanesc din Blade/config.

#### 14. Lipsesc canonical tags si sitemap

- Dovezi:
  - `public/robots.txt` are doar `User-agent` si `Disallow`, fara `Sitemap`;
  - nu exista dovada de sitemap route/file;
  - nu exista canonical in `resources/views/app.blade.php` sau in paginile Inertia inspectate.
- Impact:
  - crawlerele nu primesc semnal clar despre URL-ul canonic;
  - indexare mai slaba si risc de duplicate URL handling.
- Recomandare:
  - adauga sitemap XML si referinta in `robots.txt`;
  - adauga canonical per pagina publica.

#### 15. Acoperirea meta pentru paginile publice este incompleta si inconsistenta

- Dovezi:
  - `Article.jsx` are meta dinamice bune;
  - multe alte pagini publice au doar `Head title`, fara `description`, `og:*`, `twitter:*`;
  - `Psychologists.jsx` are titlul `Ajutor - Calming`, care nu descrie pagina.
- Impact:
  - duplicate metadata;
  - CTR scazut in cautare;
  - branding ambiguu pe pagini importante.
- Recomandare:
  - standardizeaza metadatele pentru Home, Learn, LearnCategory, Community, Psychologists, PsychologistProfile, CommunityGroup;
  - corecteaza imediat titlul paginii psihologi.

### SEO Medium

#### 16. Indexarea paginilor legale/help/tehnic nu este controlata explicit

- Dovezi:
  - paginile `Legal/Show.jsx` folosesc titluri si descrieri normale, fara `robots` sau `noindex`;
  - nu exista strat clar de decizie SEO pentru documente auxiliare.
- Impact:
  - pagini cu valoare SEO mica pot consuma crawl budget;
  - legal/help pot concura inutil cu pagini comerciale.
- Recomandare:
  - decide explicit daca `help`, `legal`, `tehnic` sunt indexabile;
  - aplica `noindex,follow` daca vrei doar accesibilitate, nu trafic organic.

#### 17. Modelul Inertia este dependent de JS pentru continutul paginilor, fara SSR

- Dovezi:
  - `resources/views/app.blade.php` foloseste shell Inertia standard;
  - nu exista indiciu de SSR.
- Impact:
  - crawlerele moderne se descurca adesea, dar discoverability si rendering pot fi mai slabe decat la HTML server-rendered.
- Recomandare:
  - pentru paginile SEO-critice, evalueaza SSR, prerender sau macar head metadata foarte bine controlate.

## Observatii de model de date

- Schema foloseste suficiente foreign keys in multe zone noi, dar modelarea este inca foarte controller/query driven.
- Exista tabele care combină responsabilitati aplicative si analitice in acelasi runtime DB: sesiuni, cache, queue, notificari, stats.
- Pentru crestere, cele mai sensibile zone sunt:
  - `assistant_messages`
  - `notifications`
  - `community_dialogue_messages`
  - `sessions`
  - `jobs` / `cache`

## Valuri de remediere

### 0-7 zile

- Scoate `api/debug/assistant` din productie.
- Corecteaza encoding-ul UTF-8 in `resources/views/app.blade.php`.
- Corecteaza titlul paginii psihologi.
- Adauga `throttle` pentru assistant guest si rutele psychologist signin/MFA resend.
- Decide si aplica `APP_DEBUG=false` in productie.
- Decide politica pentru `help/legal/tehnic`: index sau noindex.
- Repara sau aliniaza cele 3 teste picate din `ProfileTest`.

### 1-4 saptamani

- Introdu middleware dedicate pentru `superadmin` si `psychologist` cu MFA recent.
- Muta documentele de validare pe storage privat cu download autorizat.
- Adauga sitemap XML, canonical si meta standardizate pe paginile publice.
- Refactorizeaza `UserProfileBootstrapper` ca sa evite query-urile repetitive.
- Optimizeaza `NotificationService::feedFor()` si bootstrap-urile asociate.
- Restructureaza dashboard-urile cu query-uri batch si agregari SQL.

### 4+ saptamani

- Separa state-ul operational de DB-ul principal: Redis pentru cache/session/queue unde este posibil.
- Introdu retentie automata pentru sesiuni, MFA challenges, verificari email, notificari si istoricul assistant.
- Standardizeaza accesul la date pe servicii/repository sau Eloquent cu eager loading in zonele critice.
- Evalueaza SSR/prerender pentru paginile SEO-critice.

### Dupa stabilizare

- Monitorizare query time pe dashboard-uri, community si notifications bootstrap.
- Search Console / sitemap health / snippet review.
- Politica de indexare si informatie legala documentata.
